## Context

`goginx-admin` 当前以非交互式子命令为主，命令分发集中在 `cmd/goginx-admin/main.go`，通过默认部署根路径或 `-db` 打开 SQLite，再调用 `internal/admin` 完成管理员、用户、客户端、join token、代理和证书相关写入。读侧已有 `internal/adminquery`，可提供用户、客户端、仪表盘等面向页面的查询模型。

本次变更面向“本地运维工具”，重点是首次部署和小规模维护：操作者在服务器终端运行 TUI，快速完成管理员初始化、用户创建、客户端创建或客户端加入令牌生成。它不是远程管理控制台，也不替代现有浏览器管理后台。

## Goals / Non-Goals

**Goals:**

- 新增 `goginx-admin tui` 入口，默认使用与现有 CLI 一致的部署根 SQLite 路径，并继续支持 `-db` 和 `-actor`。
- 提供清晰的本地主菜单：管理员设置、用户管理、客户端配置和退出。
- 在管理员设置中优先展示现有管理员选项，支持创建管理员、更新管理员密码和启用已存在管理员。
- 在用户与客户端配置中优先使用列表选择、默认值、确认页和生成凭据，尽量避免要求操作者手写资源 ID。
- 对用户名、密码、角色、用户选择、客户端名称、凭据、join token 默认参数等字段执行提交前校验，并把服务端结构化校验错误映射回当前表单。
- 保持现有非交互式子命令兼容，不改变它们的参数或输出。
- 让 TUI 状态和渲染可测试，避免只能靠人工终端试用验证。

**Non-Goals:**

- 不实现远程 `--server` 模式，不通过 `/api/admin/graphql` 登录或维护浏览器会话。
- 不把代理、证书、审计、配额、设置和告警纳入首版 TUI 配置范围。
- 不改变 SQLite schema、管理员认证模型、客户端 credential/hash 语义或 join token 格式。
- 不提供无需本地数据库权限即可远程初始化管理员的入口。
- 不移除或弱化现有脚本友好的 CLI 子命令。

## Decisions

### TUI 作为 `goginx-admin tui` 子命令

新增 `tui` 子命令，而不是在无参数时自动进入 TUI。这样脚本和自动化仍能依赖无参数报错与现有帮助行为，操作者也能明确表达要进入交互模式。`tui` 复用通用 `-db` 和 `-actor` flag，并解析部署相对路径。

替代方案是让 `goginx-admin` 无参数启动 TUI；这会改变现有错误语义，也容易影响自动化检测。

### TUI 逻辑隔离到独立包

新增 `internal/admintui` 或等价包承载终端模型、视图状态、输入校验、服务适配器和动作编排。`cmd/goginx-admin` 只负责解析 flag、打开服务和启动 TUI。TUI 包依赖接口而不是直接耦合全局函数，便于用内存 store 或 fake 服务测试。

替代方案是把所有状态直接写进 `main.go`；当前文件已经承担多个子命令，继续堆叠交互状态会让测试和维护成本快速上升。

### 直接复用本地 admin/query 服务

TUI 读操作使用 `internal/adminquery.Service` 或 store 查询，写操作使用 `internal/admin.Service`。这样所有持久化、审计、密码哈希、凭据生成和 join token 生成继续走现有领域服务。TUI 只做交互层预校验和错误呈现，不复制业务规则。

替代方案是在 TUI 中直接操作仓储；这会绕过已有审计和校验，增加行为分叉。

### 选项优先的表单模型

用户角色使用固定选项；客户端所属用户从现有用户列表中选择；客户端凭据默认由服务生成；join token 参数先展示部署默认值，并允许操作者在确认前编辑。仅对没有合理候选项的字段使用文本输入，例如用户名、客户端显示名称和密码。

替代方案是为所有字段提供自由输入；这和“简单易用、优先选项”的目标相反，也更容易产生无效 user ID 或角色值。

### 强校验分两层执行

TUI 在提交前执行交互层校验：必填、密码确认一致、枚举选项有效、选择项仍存在、路径或 CA 文件存在、join token 地址字段非空。提交后继续接收 `internal/admin` 返回的结构化校验、冲突和不存在错误，并在当前表单或确认页展示，不把错误吞成通用失败。

替代方案是只依赖服务层报错；那会让用户走完整提交后才发现基础输入错误，TUI 的体验价值会变薄。

### 客户端配置使用默认快速向导

首版客户端配置提供一个默认快速向导：选择所属用户、填写客户端名称、确认 join token 默认参数，然后创建客户端和加入令牌。向导成功后在结果页展示一次性 token；同时保留“仅创建客户端凭据”作为次级路径，用于不需要 enrollment 流程的本地配置场景。

替代方案是把“创建客户端凭据”和“创建 join token”放成两个同级主动作；这虽然边界清晰，但会让首次接入路径多一次选择，不符合快速配置目标。

### 用户和客户端维护包含禁用与删除

首版 TUI 在用户管理和客户端配置中纳入禁用、启用和删除动作。破坏性动作必须通过现有资源列表选择目标，并在写入前展示摘要确认；删除动作还要复用服务层冲突保护，例如客户端存在启用代理时拒绝删除。为降低误操作，删除确认应比普通启用/禁用确认更强，例如要求二次确认或输入资源 ID。

替代方案是首版只做创建；这会让本地运维工具无法处理常见的账号停用、客户端下线和误建资源清理。

### 使用成熟 Go TUI 依赖

实现优先采用 Bubble Tea 生态的事件循环、列表、输入和样式组件，保持 Windows/Linux 终端兼容。依赖应只进入 admin 二进制，不影响 server/client 运行时路径。

替代方案是手写原始终端控制；这会把精力耗在键盘事件、焦点、窗口尺寸和渲染细节上，不利于快速交付稳定交互。

## Risks / Trade-offs

- TUI 依赖会增加 `goginx-admin` 构建体积 -> 仅在 admin 命令中引用，保持 server/client 不引入该依赖。
- 终端环境可能不支持交互式控制或重定向 stdin/stdout -> 进入 TUI 前检测交互能力，失败时给出明确错误并提示使用非交互式子命令。
- 本地 TUI 无法展示服务端进程内实时 session/stats -> 首版明确定位为本地配置工具，不承诺运行时监控能力。
- 生成的凭据或 token 一次性展示，用户可能关闭页面后丢失 -> 结果页明确提示只显示一次，丢失后需要重新生成或轮换。
- TUI 预校验可能与服务层规则漂移 -> 预校验只覆盖基础交互规则，最终仍以 `internal/admin` 的结构化错误为准，并用测试覆盖关键错误映射。
- Windows 和 Linux 终端表现可能有差异 -> 用可测试模型覆盖状态逻辑，并在实现阶段进行至少一次 Windows PowerShell 和类 Unix 终端手动冒烟。

## Migration Plan

新增 TUI 为可选子命令，不需要数据库迁移。发布后，既有脚本继续使用原非交互式命令；运维文档增加推荐的首次本地配置路径。回滚时移除或不使用 `tui` 子命令即可，SQLite 中已创建的管理员、用户、客户端和 join token 仍是现有模型数据。
