## Context

admin 后端方向已经确定：浏览器消费同源 admin surface，包含 `POST /api/admin/login`、`GET /api/admin/session`、`POST /api/admin/logout` 和 `POST /api/admin/graphql`。dashboard、用户、客户端、代理、证书和审计的规范 GraphQL 合同也已作为后端基线对齐。此前缺失的是位于这些合同之上的前端页面模型。

如果没有页面模型设计，后续前端实现会在路由分组、shell 职责、导航深度、页面刷新行为和失败状态上做隐式决策。这些决策是横切的，会影响登录启动、受保护导航、列表/详情组合、变更刷新策略，以及每个页面如何消费同源 admin API。

本变更捕获目标前端 shell 和页面定义，并为后续实现提供稳定的结构。

## Goals / Non-Goals

**目标：**
- 为已确认管理员视图定义一个专用 admin 前端 shell。
- 定义同源 admin console 的路由层级、shell 边界、导航模型和受保护路由行为。
- 定义加载、空状态、错误、未找到和会话过期的共享页面状态语义。
- 定义每个已确认页面如何消费规范 GraphQL 合同，包括列表/详情组合边界。
- 定义页面级轮询预期，使运行时导向页面保持新鲜，同时避免全应用刷新。
- 为后续前端实现留下可直接落地的规划任务，减少框架层歧义。

**非目标：**
- 不重设计 admin 后端合同、会话模型或 GraphQL schema。
- 不扩展到配额/设置页、告警中心、更完整可观测性页、域名工作流、RBAC 重设计或普通用户自助。
- 不要求未被后端合同支持的证书详情或审计详情路由。

## Decisions

1. 使用一个专用路由 shell，并分成两个顶层路由组。
   - 决策：admin 前端组织为未认证 `login` 路由组，以及已认证应用 shell 路由组。
   - 理由：会话启动和路由守卫行为保持一致，同时避免每页重复导航、认证检查和全局错误/会话处理。

2. 首批前端保持扁平导航。
   - 决策：主 shell 导航只包含 `Dashboard`、`Users`、`Clients`、`Proxies`、`Certificates` 和 `Audit`。
   - 理由：这些是唯一已确认视图；扁平导航符合当前 admin 范围，不发明未来区域。

3. `login` 是唯一未认证页面。
   - 决策：`/login` 是唯一公开路由，所有其他 admin 路由都要求 `/api/admin/session` 成功启动。
   - 理由：后端合同是管理员专用且同源；单一未认证入口让路由守卫保持简单。

4. 受保护路由先启动会话，再根据状态重定向。
   - 决策：受保护路由先通过 `/api/admin/session` 解析会话，缺失或过期时重定向到 `/login`，登出先清理 shell 状态再回到登录页。
   - 理由：匹配后端会话模型，并为硬刷新、直接深链和会话中途过期提供统一路径。

5. 页面层级以列表优先，只有后端合同支持详情视图的资源才有详情路由。
   - 决策：`users`、`clients` 和 `proxies` 包含列表与详情页；`dashboard`、`login`、`certificates` 和 `audit` 在本变更中是顶层页面。
   - 理由：匹配已确认后端范围，避免为 certificates 或 audit 发明未支持详情体验。

6. 共享页面状态是路由模型的一部分。
   - 决策：shell 和页面定义显式要求加载、空状态、错误、未找到和会话过期处理，且会话过期区别于普通后端失败。
   - 理由：页面状态行为是先定义前端模型的核心原因，也能防止各页面发明不同语义。

7. 轮询保持页面级且资源感知。
   - 决策：dashboard、clients 和 proxies 使用 5 秒轮询；certificates 使用低频或动作驱动刷新；users 和 audit 使用手动或低频刷新；login 不轮询。
   - 理由：保留规范后端预期，同时使前端刷新归属明确。

8. 规范 GraphQL 合同由页面容器消费。
   - 决策：每个路由定义映射到页面容器，由页面容器拥有相关列表、详情或变更流程；login/session/logout 保持为非 GraphQL 交互。
   - 理由：保持路由归属与后端合同边界一致，避免 shell 或导航承载资源变更逻辑。

## Risks / Trade-offs

- [Risk] 框架无关设计仍可能留下部分实现决策。 -> Mitigation: 已把路由归属、页面层级、状态和轮询行为定义到不随框架改变的粒度。
- [Risk] 扁平导航在未来 admin 区域扩展后可能需要调整。 -> Mitigation: 当前只记录未来触点，不把它们升格为首批路由。
- [Risk] 实现中可能推断出未支持详情页。 -> Mitigation: 明确哪些页面有列表/详情层级，其他区域保持顶层页面。
- [Risk] 会话过期可能被误处理成普通页面错误。 -> Mitigation: 要求认证过期处理区别于校验、未找到和内部失败。

## Migration Plan

1. 使用本变更最终确定前端 shell、路由图、页面层级和状态模型。
2. 后续实现按该模型选择前端框架，并映射成实际应用代码。
3. 在构建受保护页面前，先实现登录启动和受保护 shell 行为。
4. 以 dashboard 加一个列表/详情资源对验证 shell，再推进剩余页面。
5. 用路由级和页面级覆盖验证导航、深链守卫、空/错误状态和页面级轮询。

## Open Questions

- shell 根路径可以保持 `/` 再按状态跳转，也可以把已认证入口固定到 `/dashboard`；实现 MUST 保持相同页面层级。
- 如果后端未来增加独立证书详情合同，certificates 可新增详情路由；本变更不要求该路由。
- Audit 未来可能演进为更完整可观测性导航；首批 shell 仍将其作为独立近期活动页。
