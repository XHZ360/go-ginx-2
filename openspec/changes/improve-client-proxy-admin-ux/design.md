## Context

当前系统已经支持 configless 服务端启动、客户端 join/enrollment 材料生成、admin GraphQL 资源管理，以及专用 admin-ui。现有 join 材料可以携带服务端地址，但默认值通常由调用端显式传入；这让基础部署在生成 token 时仍可能要求管理员重复填写控制通道地址。客户端管理已有创建、查看和凭据轮换合同，但缺少永久删除语义。前端已具备客户端、代理和弹窗表单，但跨资源创建和弹窗边界行为还不够明确。

这次变更跨越服务端配置、join 材料生成、admin 命令/API、SQLite 资源生命周期和 admin-ui 交互，因此需要在实现前固定默认地址来源、删除约束和前端状态传递方式。

## Goals / Non-Goals

**Goals:**

- 服务端启动时产出一个可解释、可覆盖的“确认服务地址”，供后续 join token 默认使用。
- join token 生成路径默认使用已确认服务地址，同时保留显式地址覆盖。
- 管理员可以永久删除客户端资源，并且删除行为不会绕过已存在的代理生命周期保护。
- 从客户端上下文进入创建代理时，表单自动带入 `userId` 与 `clientId`，并允许管理员通过选择器调整。
- 建立统一弹窗最大高度、滚动、token 展示和蒙版关闭语义。

**Non-Goals:**

- 不实现公网 NAT 穿透、动态 DNS、自动证书名称发现或多地址健康探测。
- 不改变控制通道协议、join token 编码格式以外的安全模型，且不记录可重放 secret。
- 不支持批量删除客户端、批量迁移代理或跨用户转移客户端。
- 不新增正向代理能力，不扩大当前代理类型范围。
- 不重写整个 admin-ui 组件库或引入新的前端 UI 依赖。

## Decisions

1. 服务地址使用“显式配置优先，自动推断兜底”

   新增或复用服务端运行时字段保存 confirmed service host/address。优先级为显式配置值、控制通道监听地址中可用于客户端连接的 host、最后是本机可用非回环 IP 或 `127.0.0.1` 本地兜底。备选方案是每次生成 join token 都要求输入地址；该方案更精确但破坏 configless 的默认体验。自动推断结果必须可见于日志或管理启动信息，方便管理员发现 NAT/反向代理场景下需要覆盖。

2. Join token 生成只消费已确认默认值，不在 token 生成处重新做网络推断

   地址确认属于配置/启动责任，join 生成属于资源管理责任。这样 CLI、GraphQL 和未来 UI 入口可以共享同一个默认值，并且测试可以分别覆盖“确认默认值”和“生成 token 使用默认值”。显式传入 server address、TLS address、server name 或 enrollment URL 时仍覆盖默认值。

3. 客户端删除采用永久删除加运行时保护

   删除客户端时必须删除或失效客户端凭据、join/enrollment 残留材料和允许级联的从属持久化资源；删除后列表、详情、认证和 join 消费都不能再解析该客户端。若客户端仍有关联的启用代理，删除请求返回结构化冲突，要求管理员先禁用代理。备选的软删除会保留歧义状态，不满足“完全删除”；无条件级联启用代理会破坏运行时监听和审计可解释性。

4. 客户端到代理创建使用 URL/路由状态承载默认上下文

   从 client 列表或详情触发创建代理时，前端导航到 proxies 创建入口并携带 `clientId` 和 `userId`。创建表单初始化选择器默认值，但不锁死选择器；当候选项尚未加载时，以 ID fallback 保留当前值。相比全局 store，这种方式能支持刷新、深链和登录后恢复。

5. 弹窗滚动和关闭语义放在共享 Dialog/Modal 基础组件

   弹窗容器设置视口相关最大高度，主体区域滚动；token/secret 展示组件设置最大 3 行高度并独立滚动。蒙版关闭记录 pointer/mouse down 起点，只有 down 和 up/click 都发生在蒙版本身时才触发关闭。这样每个表单无需重复实现边界逻辑。

## Risks / Trade-offs

- [Risk] 自动推断地址在 NAT、公网负载均衡或容器网络中可能不是客户端可达地址。→ 显式配置和生成 token 覆盖必须优先，并在日志/UI 中展示当前默认来源。
- [Risk] 删除客户端时从属数据级联不完整会留下不可见孤儿记录。→ 数据库外键、服务层事务和删除后查询测试必须同时覆盖客户端、凭据、代理、证书和 join/enrollment 相关记录。
- [Risk] 阻止带启用代理的客户端删除会让“完全删除”多一步。→ 这是为了保持现有代理禁用/删除生命周期，错误需要返回可操作的冲突信息。
- [Risk] URL 中携带默认 `clientId`/`userId` 时，候选项加载失败可能导致选择器显示不完整。→ 表单保留 ID fallback 并阻止提交不一致的 user/client 组合。
- [Risk] 改造共享弹窗可能影响所有管理表单。→ 使用组件级测试和关键页面交互测试覆盖创建、确认、token 展示和蒙版拖放释放场景。

## Migration Plan

1. 增加服务端确认地址配置字段/运行时状态，保持旧配置文件兼容；未配置时使用自动推断默认值。
2. 更新 join 生成服务、admin CLI 和 GraphQL resolver，使缺省地址来自确认服务地址，显式输入保持最高优先级。
3. 增加客户端永久删除 command、GraphQL mutation 和 SQLite 存储能力，先拒绝仍有启用代理的客户端。
4. 更新 admin-ui 客户端页面、代理创建表单和路由参数处理，补齐用户/客户端选择器默认值与修改能力。
5. 统一 Dialog/Modal 和 token/secret 展示组件的高度、滚动和蒙版关闭行为。
6. 发布时无需数据迁移；回滚时保留已删除客户端不可恢复这一事实，其他新增 UI 行为随前端回滚消失。

## Open Questions

- 是否需要把“确认服务地址”的显式覆盖字段暴露为环境变量，方便无 JSON 配置的服务单元配置公网域名？本变更至少要求配置/启动路径具备覆盖能力，具体入口可在实现中按现有配置模式落地。
