# GoGinX SDK 设计方案

## 1. 需求定义

构建一个 Go library SDK，供第三方应用（安卓、Web 等）调用，用于从公网安全访问服务器上暴露的网络服务。

**核心能力**：
1. 使用有效 credential（复用客户端凭据模型：client_id + credential）登录
2. 从服务端获取当前客户端可访问的代理列表（用户所有客户端定义并激活的 proxy）
3. 使用与现有 goginx-client 类似的通信机制转发网络消息
4. 两种使用模式：**直接连接模式**（应用代码直接打开流）和**本地代理模式**（SDK 在本地暴露 SOCKS5/HTTP 代理入口）

```
│                                                         │
│  ┌──────────────────────────────────────────────────┐   │
│  │ Control Listener (现有)                            │   │
│  │  + SDK 流接受 (新增)                               │   │
```

**关键**：SDK 通过控制通道打开流 → 服务端桥接到 goginx-client → goginx-client 连接实际目标。流量全程通过控制通道多路复用，不暴露额外端口。

## 5. 控制通道协议扩展

### 5.1 现有协议复用

SDK 复用现有协议：

- `AuthRequest` / `AuthResponse` — 认证
- `ProxySnapshot` — 获取代理列表（⚠️ 见 §13 B1：现有 `ProxySnapshot` 按 client 作用域，无法满足「按 user 拿全量 proxy」，需扩展）
- `Heartbeat` — 保活
- `OpenStream` — 打开到代理的流

> ⚠️ **评审修正**：原文「不引入新的消息类型」不成立。要满足需求#2（用户所有客户端的 proxy），必须新增按 user 的查询与消息。详见 §13 B1。

### 5.2 客户端侧 OpenStream

`ClientConn` 新增 `OpenStream` 方法：

```go
// internal/control/transport.go

// OpenStream 在控制通道上打开一个新的多路复用流
func (client *ClientConn) OpenStream(ctx context.Context) (io.ReadWriteCloser, error) {
    if client.conn != nil {
        // QUIC: 打开新的 QUIC 流
        stream, err := client.conn.OpenStreamSync(ctx)
        if err != nil {
            return nil, err
        }
        return wrapProxyStream(stream), nil
    }
    if client.mux != nil {
        // TCP+TLS: 通过 mux 打开新的流
        return client.mux.OpenStream(ctx)
    }
    return nil, errors.New("no control connection available")
}
```

### 5.3 服务端侧 SDK 流处理

服务端在 `handleControl` 中新增 goroutine 接受 SDK 发起的流：

```go
// internal/control/transport.go — handleControl 修改

func (server Server) handleControl(ctx context.Context, stream io.ReadWriteCloser, opener session.StreamOpener) {
    // ... 现有认证和 ProxySnapshot 逻辑 ...

    // 新增：启动接受 SDK 流的 goroutine
    sdkCtx, sdkCancel := context.WithCancel(ctx)
    defer sdkCancel()
    go server.acceptSDKStreams(sdkCtx, opener, result.Client.ID, result.User.ID)

    // 现有心跳循环
    server.handleHeartbeats(stream)
}

// acceptSDKStreams 接受并处理 SDK 发起的流
func (server Server) acceptSDKStreams(ctx context.Context, opener session.StreamOpener, clientID, userID string) {
    for {
        stream, err := acceptStream(ctx, opener)
        if err != nil {
            return
        }
        go server.handleSDKStream(stream, clientID, userID)
    }
}
```

`acceptStream` 需要根据 opener 类型（QUIC 或 TCP+TLS mux）接受流：

```go
func acceptStream(ctx context.Context, opener session.StreamOpener) (io.ReadWriteCloser, error) {

    // TCP+TLS: opener 是 tcpTLSMux，已有 AcceptStream
    // 需要在 StreamOpener 接口中新增 AcceptStream 方法，
    // 或使用类型断言
```

### 5.4 SDK 流桥接

```go
// handleSDKStream 处理 SDK 发起的流请求
func (server Server) handleSDKStream(stream io.ReadWriteCloser, sdkClientID, sdkUserID string) {
    defer stream.Close()

    // 1. 读取 SDK 发送的 OpenStream 消息
    envelope, err := ReadMessage(stream)
    if err != nil || envelope.Type != MessageOpenStream {
        return
    }
    request, err := DecodePayload[OpenStream](envelope)
    if err != nil {
        return
    }

    // 2. 查找代理配置
    // ⚠️ 评审修正：Server 无 Store 字段，正确路径是 server.Authenticator.Store（见 §13 B3）
    proxy, err := server.Authenticator.Store.Proxies().ByID(context.Background(), request.ProxyID)
    if err != nil {
        return
    }

    // 3. 验证 SDK 用户有权访问该代理
    // ⚠️ 评审修正：还需校验 proxy.Status == 已启用（见 §13 M4）
    if proxy.UserID != sdkUserID || proxy.Status != domain.ProxyEnabled {
        return
    }

    // 4. 查找代理所属客户端（provider）的活跃会话
    // ⚠️ 评审修正：proxy.ClientID 必须是 provider 的 client_id，且该 session 必须是 provider 角色，
    //    不能是 SDK consumer 自身，否则桥接自环（见 §13 B2）。
    latest, ok := server.Sessions.Latest(proxy.ClientID)
    if !ok || latest.StreamOpener == nil {
        return
    }

    // 5. 打开到 goginx-client 的流
    clientStream, err := latest.StreamOpener.OpenStream(context.Background())
    if err != nil {
        return
    }
    defer clientStream.Close()

    // 6. 转发 OpenStream 消息到 goginx-client（使用代理的实际目标）
    // ⚠️ 评审修正：Kind 由 proxy.Type 派生，不透传 SDK 的 request.Kind（见 §13 m1）
    if err := WriteMessage(clientStream, MessageOpenStream, OpenStream{
        Kind:         kindFromProxyType(proxy.Type),
        ProxyID:      proxy.ID,
        ConnectionID: request.ConnectionID,
        TargetHost:   proxy.TargetHost,
        TargetPort:   proxy.TargetPort,
    }); err != nil {
        return
    }

    // 7. 双向桥接 SDK 流 ↔ goginx-client 流
    tunnel.CopyBidirectional(stream, clientStream)
}
```

## 6. StreamOpener 接口扩展

当前 `session.StreamOpener` 只有 `OpenStream`。SDK 需要服务端能从同一连接接受流。需要扩展接口：

```go
// internal/session/manager.go

type StreamOpener interface {
    OpenStream(ctx context.Context) (io.ReadWriteCloser, error)
}

// 新增：支持接受流的 StreamOpener
type StreamAcceptor interface {
    StreamOpener
    AcceptStream(ctx context.Context) (io.ReadWriteCloser, error)
}
```

QUIC 和 TCP+TLS mux 都天然支持 `AcceptStream`。服务端在 `acceptSDKStreams` 中通过类型断言检查 opener 是否实现了 `StreamAcceptor`。

## 7. 认证模型

### 7.1 复用客户端凭据

SDK 使用与 goginx-client 相同的认证模型：
- 管理员通过 Admin UI/API/CLI 创建一个 "client" 记录
- 获取 client_id + credential
- SDK 用这对凭据连接控制通道

### 7.2 代理访问权限

> ⚠️ **评审修正（核心矛盾，见 §13 B1/B2）**：
> - 现有 `ProxySnapshot` 走 `ByClientID`，返回的是**连接的那个 client 自己的** proxy，**不是** user 的全量 proxy。
> - SDK 若复用 provider 的 client_id，会 replace 掉 provider 的 session 并造成桥接自环；若用独立 client_id，则该 client 没有 proxy，`ProxySnapshot` 为空。
> - 因此本节的目标（SDK 访问该 user 的所有 proxy）**当前实现无法满足**，需按 §13「修正方向」1、2 落地：新增 `ByUserID` 查询 + 新消息，并在 session 层区分 consumer/provider。

目标（需 §13 修正后成立）：
- 管理员为 SDK 用户创建**独立的 consumer client** → SDK 能访问该 user 的所有已启用代理
- 如果需要限制 SDK 只能访问部分代理，需要在代理级别新增访问控制（建议提前，消费入口比 provider 更敏感）

### 7.3 SDK vs goginx-client 的区别

| 维度 | goginx-client | SDK |
|------|---------------|-----|
| 认证 | client_id + credential | 相同 |
| 代理列表 | ProxySnapshot | 相同 |
| 入站流 | AcceptStream → 拨号本地目标 | 不处理入站流 |
| 出站流 | 不发起 | OpenStream → 服务端桥接 |
| 心跳 | SendHeartbeat | 相同 |

## 8. 本地代理模式

> ⚠️ **评审修正（概念冲突，见 §13 M1）**：goginx 每个 proxy 的 target 是**固定**的（服务端注入 `proxy.TargetHost/TargetPort`，SDK 不能指定目标）。而 SOCKS5/HTTP CONNECT 的本质是「客户端指定任意目标」。如果「把所有连接转发到单一 proxyID」，SOCKS5/CONNECT 握手请求的目标地址会被丢弃——这不是真正的 SOCKS5 代理，只是「用 SOCKS5 外壳暴露一个固定远端服务」。
>
> 需二选一：
> 1. **固定目标隧道**：明确本地入口只对应一个 proxy 的固定 target（下方示例即此语义，建议改名如 `ExposeLocal`）。
> 2. **真 SOCKS5**：本地代理按请求的 host:port **匹配** user 名下对应的 proxy（需 §13 修正后的 by-user proxy 列表），或让 SDK 传目标（需重估「target 由服务端注入」的安全性）。

SDK 的 `StartLocalProxy` 在本地启动一个 SOCKS5/HTTP CONNECT 代理，将所有连接转发到指定的远程代理：

```go
// sdk/local_proxy.go

// StartLocalProxy 在 addr 上启动本地 SOCKS5 代理
// 所有连接通过控制通道转发到 proxyID 对应的远程代理
func (c *Client) StartLocalProxy(ctx context.Context, addr string, proxyID string) error {
    listener, err := net.Listen("tcp", addr)
    if err != nil {
        return err
    }
    defer listener.Close()

    for {
        conn, err := listener.Accept()
        if err != nil {
            if ctx.Err() != nil {
                return ctx.Err()
            }
            continue
        }
        go c.handleLocalConn(ctx, conn, proxyID)
    }
}

func (c *Client) handleLocalConn(ctx context.Context, conn net.Conn, proxyID string) {
    defer conn.Close()

    // 自动检测 SOCKS5 vs HTTP CONNECT
    // SOCKS5: 首字节 0x05
    // HTTP: 首字节 C/G/P (CONNECT/GET/POST)

    // 打开到远程代理的流
    remote, err := c.Dial(ctx, proxyID)
    if err != nil {
        return
    }
    defer remote.Close()

    // 对 SOCKS5: 完成握手后双向转发
    // 对 HTTP CONNECT: 完成握手后双向转发
    // 对直接 TCP: 直接双向转发
}
```

## 12. SDK 使用示例

```go
package main

import (
    "context"
    "fmt"
    "io"
    "log"
    "os"

    "github.com/simp-frp/go-ginx-2/sdk"
)

func main() {
    client := sdk.New(sdk.Config{
        ServerAddress:    "control.example.com:8443",
        ServerName:       "go-ginx-control.local",
        ServerCAFile:     "data/certs/server-ca.crt",
        ClientID:         "sdk-client-1",
        Credential:       "secret",
        AllowedProtocols: []string{"quic", "tcp_tls"},
    })

    ctx := context.Background()
    if err := client.Connect(ctx); err != nil {
        log.Fatal(err)
    }
    defer client.Close()

    // 获取可访问的代理列表
    proxies, err := client.Proxies(ctx)
    if err != nil {
        log.Fatal(err)
    }
    for _, p := range proxies {
        fmt.Printf("Proxy: %s (%s) %s:%d → %s:%d\n",
            p.Name, p.Type, p.EntryHost, p.EntryPort, p.TargetHost, p.TargetPort)
    }

    // 直接连接模式
    conn, err := client.Dial(ctx, "tcp-proxy-1")
    if err != nil {
        log.Fatal(err)
    }
    defer conn.Close()
    conn.Write([]byte("hello"))
    buf := make([]byte, 1024)
    n, _ := conn.Read(buf)
    fmt.Println(string(buf[:n]))

    // 本地代理模式
    go client.StartLocalProxy(ctx, ":1080", "http-proxy-1")
    // 然后用 curl -x socks5://127.0.0.1:1080 访问
}
```

## 13. 设计评审与修订（2026-07-07）

已对照实际代码（`internal/control`、`internal/session`、`internal/store`、`internal/domain`）逐条核对。整体方向成立，但存在若干阻断级问题，其中 B1/B2 是自相矛盾的核心缺口。以下按严重程度列出，均附代码证据。

### 🔴 阻断级

**B1 — `ProxySnapshot` 作用域 = 单个 client，与需求#2（用户全量 proxy）冲突**
- `sendProxySnapshot` 走 `proxyRepository.ByClientID(ctx, clientID)`（`transport.go:380`），只返回连接的那个 client 自己的 proxy。
- `ProxyRepository` 接口**没有 `ByUserID`**（`store.go:131-145`，仅 `ByID / ByClientID / List / EnabledByType / By*Entry...`）。
- ⟹ §5.1 「不引入新消息」不成立。需新增 `ProxyRepository.ByUserID`（`where user_id=?`）+ 新消息（或给 `ProxySnapshot` 加 scope 字段）。

**B2 — SDK 用哪个 `client_id` 都是死路**
- 复用 provider 的 client_id：`Manager.Register` 按 client_id 替换 latest session（`manager.go:106-115`）⟹ SDK 一连上就 replace 掉 provider，provider 掉线；且 `Sessions.Latest(proxy.ClientID)` 会取到 SDK 自己，`StreamOpener` 指回自身 ⟹ 桥接自环。
- 独立 client_id：该 client 无 proxy ⟹ `ProxySnapshot` 空 ⟹ 回到 B1。
- ⟹ 必须在 **session 层区分 consumer / provider 角色**（`RegisterInput` 加 Role 或独立注册表）：consumer 不 replace provider、不误标 Online、`Latest` 不取错对象。

**B3 — `server.Store` 字段不存在**
- §5.4 用 `server.Store.Proxies()`，但 `Server` 无 `Store` 字段（`transport.go:34-40`）。正确路径 `server.Authenticator.Store.Proxies()`（`transport.go:376`）。`ByID` 存在（`store.go:133`）。（已在 §5.4 内联修正）

### 🟠 中等

**M1 — 本地代理模式与「固定 target」模型冲突**（已在 §8 内联修正）。`handleSDKStream` 固定注入 `proxy.TargetHost/TargetPort`，SDK 无法选目标，与 SOCKS5/CONNECT「任意目标」语义冲突。

**M2 — `acceptSDKStreams` 对所有连接无条件开启，缺角色/权限层**
- §5.3 对每个连接都起 `acceptSDKStreams`。provider 连接上永远阻塞（provider 不主动 open 流），无害但浪费；更关键是无 consumer/provider 区分，任何认证 client 只要其 user 拥有该 proxy 即可桥接。安全模型需显式说明，per-proxy ACL 建议提前。

**M3 — TCP+TLS mux 时序 & `acceptStream` 仍是 TODO**
- TCP+TLS 下 mux 在 `ProxySnapshot` 之后才 `Start`（`transport.go:332-346`），`mux.AcceptStream` 依赖 `readLoop`（Start 后才有）⟹ `acceptSDKStreams` 必须放在 `mux.Start()` 之后。
- `quicStreamOpener` 当前不暴露 `AcceptStream`（`transport.go:64-74`），`StreamAcceptor` 需给它补方法；§5.3 的 `acceptStream` 尚未实现。
- ✅ 好消息：mux 协议层天然双向（server 开奇数、client 开偶数，`handleOpen` 按 parity 校验，`mux.go:135`），QUIC 亦支持 control 流之后继续 `AcceptStream`，故机制可行。

**M4 — 未校验 proxy 是否 enabled**（已在 §5.4 内联修正）。原 `handleSDKStream` 只查 `proxy.UserID`，需加 `proxy.Status == 已启用`。

### 🟡 次要

- **m1** `OpenStream.Kind` 应由 `proxy.Type` 派生，不透传 SDK（provider 按 Kind 分流：`transport.go:533-541`）。（已在 §5.4 内联修正）
- **m2** `ProxyInfo` 暴露 `EntryHost/EntryPort` 对纯消费者意义不大（SDK 走控制通道桥接，不用公网入口），易误导；`Type` 的 "https" 需与 `domain.ProxyType` 实际取值对齐。
- **m3** SDK 桥接流未纳入 provider `ActiveStreams` 心跳统计。
- **m4** SDK 连接会把该 client 标记 `Online`（`transport.go:347`）并占一个 session，把消费者混入「客户端在线」状态（与 B2 同源）。

### ✅ 站得住的部分

- 控制通道多路复用桥接、不额外开端口的大方向正确。
- mux 双向 / QUIC 二次 Accept 均成立，机制层面可行。
- `domain.Proxy` 字段够用（`UserID/ClientID/TargetHost/TargetPort` 齐全，`protocol.go:410`）。
- 复用 `tunnel.CopyBidirectional`（`tunnel.go:52`）、复用认证 —— 合理。

### 修正方向（落地顺序）

1. **补 by-user proxy 查询**：`ProxyRepository.ByUserID` + 新消息（或扩展 `ProxySnapshot` 语义/scope），让 consumer 拿到 user 全量 enabled proxy。（解 B1）
2. **session 层区分 consumer/provider**（Role 字段或独立表）：consumer 不 replace provider、不误标 Online；`handleSDKStream` 按 `proxy.ClientID` 找 provider 且排除 consumer 自身。（解 B2）
3. 修 `server.Authenticator.Store` 路径；`handleSDKStream` 补 `proxy.Status` 校验、`Kind` 由 `proxy.Type` 派生。（B3/M4/m1）
4. 重新定位本地代理模式：固定目标隧道 or 按目标匹配 proxy。（M1）
5. `StreamAcceptor` 给 `quicStreamOpener` 加 `AcceptStream`；`acceptSDKStreams` 放到 `mux.Start()` 之后；补齐 `acceptStream`。（M3）
6. per-proxy consumer ACL 提前。（M2）

> **实施计划调整**：§9 的 Phase 1 需前置「store 层 `ByUserID` + session 角色区分 + 新消息」，否则 Phase 2 的 `Proxies()` 返回空，SDK 无可用代理。

## 14. 资源限制与 DoS 防护

### 14.1 设计前提（premise）

1. **容忍 consumer 压垮 provider**。二者同属一个 user，consumer 只能访问 `proxy.UserID == sdkUserID` 的代理（§5.4 step3），故爆炸半径天然锁定在**单一用户**内。因此**不做** provider 保护类措施（不预留 provider 流预算、不为保护 provider 而限流/熔断、不为 provider 单独降低 `MaxIncomingStreams`）。
2. **不接受服务端被冲垮**。服务端的全局资源（goroutine、内存、连接、文件描述符）必须有硬上限，任何单一用户都不能耗尽。

### 14.2 核心原则：每用户资源配额 + 全局硬顶

给每个 user（或每条 consumer 连接）分配一份**有界的服务端资源切片**：
- 配额内，consumer 可以随意打满、乃至饿死自己的 provider —— 容忍（premise 1）。
- 所有切片之和受**全局硬顶**约束；越界即**主动拒绝/reset 新流**（load shedding），损害只留在越界用户自己的切片里（premise 2）。

关键点：premise 1「容忍 provider 被压垮」**不等于**「允许服务端为被阻塞的 provider open 无限等待」。服务端必须始终限住自己的 footprint。

### 14.3 具体机制

| # | 机制 | 目的 | 落点 |
|---|------|------|------|
| R1 | **全局并发桥接流硬顶** `maxBridgedStreamsGlobal`（信号量）。达到后新 SDK 流立即 reset。 | 封顶全局 goroutine/内存 | `acceptSDKStreams`：spawn `handleSDKStream` 前 `TryAcquire`，失败即 reset；流结束 `Release` |
| R2 | **每 user / 每 consumer 连接并发上限**（如 `maxBridgedStreamsPerUser`）。 | 防单一用户吃满全局配额 | `handleSDKStream` 入口按 `sdkUserID` 计数 |
| R3 | **provider `OpenStream` 带超时 context**，禁用 `context.Background()`。超时即 reset SDK 流并释放。 | provider 被打满时**释放服务端** goroutine/流（不是保护 provider） | §5.4 step5 改 `context.WithTimeout` |
| R4 | **流 open 速率限制**（每连接令牌桶）。 | 挡 open/close 抖动导致的 goroutine churn | `acceptSDKStreams` accept 后过限流器 |
| R5 | **并发控制连接数上限**（全局 + 每 client_id），QUIC 侧补 admission 信号量（类比 TCP+TLS 的 `handshakes` chan，`transport.go:98`）。 | QUIC 无连接准入，防多连接绕过每连接流上限 | `Listener.handleConn` 前置准入 |
| R6 | **session 被 replace 时主动关闭旧物理连接**。 | 防旧连接残留占用流预算/内存 | `Manager.Register` 返回 replaced 后关闭其 `StreamOpener`/conn |
| R7 | **背压优先于 spawn**：`acceptSDKStreams` 在拿到 R1/R2 slot **之前不 accept**（或 accept 后立即 reset）。 | 让 QUIC/mux 流控天然回压到 consumer | accept 循环结构 |

### 14.4 非目标（premise 1 明确排除）

- ❌ 为保护 provider 预留/隔离其流预算
- ❌ 为保护 provider 的 consumer 侧限流、熔断、公平调度
- ❌ consumer 与 provider 使用不同 `MaxIncomingStreams` 以护 provider

> R3 看似护 provider，实为护**服务端**：provider 阻塞时不让服务端 goroutine 无限挂住。这与 premise 1 不冲突。

### 14.5 参数建议

- `maxBridgedStreamsGlobal`：按内存预算反推（每流 ≈3 goroutine + 2×32KB 缓冲 ≈ 单流几十 KB~百 KB 量级），例如按可用内存留一档，先给保守值再压测调。
- `maxBridgedStreamsPerUser`：`global / 预期并发用户数` 或固定分数（如 global 的 10~25%）。
- provider `OpenStream` 超时：数秒级（如 3~5s）。
- open 速率：每连接每秒数十~百级令牌，突发桶适度。
- 全局/每 client_id 连接数：小整数（如每 client_id ≤ 4）。

### 14.6 实施计划补充

上述 R1~R7 归入 §9 **Phase 1（控制通道扩展）**，与 `acceptSDKStreams`/`handleSDKStream` 同批落地——否则一旦通了桥接，服务端即处于无保护状态。压测（大量并发流 + open/close 抖动 + provider 阻塞）纳入 §11 验证。
