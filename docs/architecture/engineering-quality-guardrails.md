# 工程质量分层防线

这份文档记录从外部服务集成、证书生命周期、Admin API/UI 故障中沉淀出来的通用经验。它不只适用于 Cloudflare Origin CA，也适用于所有同时包含“本地状态、外部提供方调用、后台操作界面、异步生命周期”的功能。

核心目标是：把可预期的失败变成可解释、可测试、可恢复的业务状态，而不是让它们滑落成 `internal server error`。

## 适用场景

- 调用外部提供方 API 创建、更新、删除或轮换资源。
- 本地数据库、文件系统或缓存需要保存外部资源的生命周期状态。
- 外部调用可能非幂等，或者失败时仍可能已经产生远端副作用。
- Admin API、GraphQL、CLI 或 UI 会把生命周期状态暴露给操作者。
- 功能涉及证书、密钥、令牌、凭据等敏感材料。

## 一、设计层：先定义生命周期边界

外部提供方调用必须被视为不稳定、可能非幂等、可能出现部分成功的边界。设计时应先回答：

- 本地状态是在外部调用前写入，还是外部调用后写入？
- 如果外部调用失败，本地是否已经留下占位记录？
- 如果外部调用看似失败，远端是否可能已经创建了资源？
- 首次创建失败和已有可用资源的续期、轮换失败，是否应该进入不同状态？
- 是否存在可用于重试、查询、去重或清理的远端资源 ID？

推荐语义：

- 首次创建还没有可用材料时，外部调用失败应清理空占位，或明确标记为不可用且不会被运行时选中。
- 续期、轮换已有可用材料时，失败应保留旧材料继续服务，同时记录失败原因和下一次可重试时间。
- 删除或替换远端资源时，应明确补偿策略：远端删除失败、本地删除失败、文件删除失败分别如何处理。
- 生命周期状态字段应能表达成功、进行中、失败、可重试、已废弃等状态，不要只依赖空字符串或错误文本推断。

## 二、服务与 API 层：把错误变成契约

服务层要把错误分成操作者和调用方能理解的类别：

- `validation`：输入不合法，本地即可判断。
- `conflict`：请求和当前资源状态、远端约束或唯一性规则冲突。
- `provider_failed`：外部提供方拒绝或返回失败。
- `provider_unavailable`：外部提供方超时、网络不可达或临时不可用。
- `persistence_failed`：本地数据库或文件系统写入失败。
- `internal`：真正无法归类、需要开发者排查的异常。

API 和 GraphQL 边界应遵循：

- 可预期的外部提供方错误不应映射成通用 `internal server error`。
- 返回给 UI 的错误应足够可操作，但不能泄露密钥、令牌、证书私钥、完整请求体或敏感响应体。
- `lastError` 可以记录经过清洗的提供方状态码、错误码和摘要信息。
- 成功响应和查询模型应暴露生命周期元数据，例如状态、上次失败、远端资源 ID、更新时间。
- 调用方需要能区分“没有创建成功”和“创建成功但后续步骤失败”。

## 三、持久化与补偿层：让部分失败可恢复

凡是“本地写入 + 外部调用 + 文件写入”组合，都要显式设计事务边界和补偿动作。

检查点：

- 本地事务是否覆盖了应该原子完成的写入？
- 外部调用无法放进本地事务时，是否有补偿或对账机制？
- 如果外部创建成功但本地写入失败，是否能通过远端 ID、名称或审计日志找回并处理？
- 如果本地占位创建成功但外部创建失败，是否会留下空记录、坏状态或运行时误用的记录？
- 如果证书或密钥文件写入失败，数据库状态是否会错误显示为可用？

证书和密钥类材料的额外约束：

- 私钥不得进入 API 响应、日志、错误消息和 `lastError`。
- 数据库中保存路径、状态、远端 ID 和必要元数据，避免保存非必要敏感材料。
- 轮换失败时优先保留当前可用材料，避免把运行时从可用状态推入不可用状态。

## 四、Admin UI 层：让操作者看见真实状态

后台界面不只是提交表单，也要承接生命周期的可观察性。

UI 应覆盖：

- 创建、续期、删除、失败重试等操作的明确反馈。
- 外部提供方错误的可读提示，而不是只显示“服务器内部错误”。
- 资源状态、上次错误、更新时间、远端 ID 或脱敏摘要。
- 操作中、成功、失败、可重试、不可操作等按钮状态。
- 重复提交防护，但不能用禁用按钮掩盖真实错误。

## 五、测试层：按失败面建立矩阵

涉及外部提供方的功能，测试不能只覆盖成功路径。每个生命周期操作至少应考虑下面的矩阵：

| 场景 | 需要验证 |
| --- | --- |
| 首次创建成功 | 本地状态、远端 ID、文件路径、查询模型和 UI 状态一致。 |
| 外部提供方返回业务错误 | API 返回可消费错误码，`lastError` 脱敏且可操作，不产生通用 500。 |
| 外部提供方超时或网络失败 | 状态可重试，不泄露敏感信息，不误标记为成功。 |
| 首次创建失败且没有可用材料 | 空占位被清理，或被标记为运行时不可选。 |
| 已有材料续期或轮换失败 | 旧材料保留可用，新失败状态和错误信息可查询。 |
| 本地数据库写入失败 | 不返回假成功，存在远端副作用时有补偿或对账线索。 |
| 证书、密钥或配置文件写入失败 | 数据库不会显示为可用，敏感内容不进入错误信息。 |
| API/GraphQL 错误映射 | 预期错误映射到 `validation`、`conflict`、`provider_failed` 等契约。 |
| Admin UI 错误展示 | 操作者能看到脱敏、可操作的失败原因。 |
| Secret safety | 日志、API 响应、测试断言都不包含私钥、令牌或完整敏感载荷。 |

测试分层建议：

- 领域或服务单元测试覆盖生命周期状态、补偿和错误分类。
- API 测试覆盖 GraphQL/HTTP 错误映射和响应结构。
- UI 测试覆盖失败提示、状态展示和重试入口。
- E2E 或 smoke 测试覆盖真实进程启动、配置加载和关键操作链路。

## 六、发布与运行校验层：把验证写进流程

发布前的最低检查：

```powershell
$env:CGO_ENABLED="0"
go test ./...
```

如果改动涉及 Admin UI，还应执行对应前端测试和构建命令。具体命令以 `admin-ui` 当前脚本为准。

部署后 smoke 检查：

- 确认服务进程已加载新二进制。
- 查看服务状态和最近日志，确认没有启动期错误。
- 使用 Admin API/UI 执行一次只读查询，确认会话和 GraphQL 正常。
- 对本次改动涉及的关键生命周期执行最小化验证，例如创建失败、查询状态、重试入口。

对证书类资源，可增加异常状态巡检：

```sql
select *
from managed_certificates
where cert_file = ''
  and key_file = ''
  and cloudflare_certificate_id = ''
  and status in ('issue_failed', 'renewal_failed');
```

## 七、评审清单：合并前逐项过一遍

- 这段代码是否在调用外部 API 前写入了本地状态？
- 如果外部 API 失败，本地状态会被清理、标记还是保留？
- 如果外部 API 成功但本地后续步骤失败，是否有补偿、对账或人工处理线索？
- 首次创建失败和已有资源续期、轮换失败是否被分别测试？
- 提供方错误是否映射成调用方能处理的 API 错误，而不是通用 500？
- `lastError`、日志和 UI 是否只包含脱敏后的必要信息？
- UI 是否展示真实状态和失败原因？
- 发布流程是否包含后端测试、前端测试或构建、部署后 smoke 检查？
- 是否有用于发现历史异常状态的查询或巡检办法？

## 八、事故后沉淀：从个案回收到通用规则

每次问题修复后，应把经验拆成三类沉淀：

- 代码修复：修正当前 bug，并补充防回归测试。
- 流程修复：把遗漏的测试、发布校验或运行巡检纳入固定流程。
- 文档修复：把可复用经验写入通用文档，而不是只写在一次性的聊天记录或事故说明里。

当某类问题第二次出现时，应优先检查测试矩阵和评审清单，而不是只继续在局部打补丁。
