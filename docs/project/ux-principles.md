# 项目 UX 原则

## 定位

本项目用于管理反向代理、Domain、客户端访问、证书、凭据和运行时状态。UX 的首要目标不是减少按钮数量，而是让操作者在高影响操作中准确理解当前状态、风险、责任边界和下一步。

本文件定义跨功能的长期体验约束。具体业务流程、字段和验收口径仍以 `requirements/` 为准；进行中的实现差异记录在 `changes/`。

## 一、以真实系统状态驱动界面

- 界面显示、可用动作和默认值必须基于后端真实状态，不能以过期缓存、乐观假设或其他资源的状态代替。
- 只有操作在服务端实际可执行时才允许提交；不能执行时应保留入口、禁用操作并说明原因和恢复路径。
- 前端预检查只用于引导，服务端必须在执行前重新校验所有权限、依赖和运行条件。
- 多维状态必须分别展示，不把“是否可服务”“最近操作结果”“外部 provider 状态”压缩成含糊的单一标签。

## 二、将可预见失败前置化

- 输入校验、权限缺失、依赖资源缺失、服务端配置未就绪和已知冲突应在提交前提示。
- 错误提示必须说明“发生了什么、为什么、谁能修复、下一步做什么”。
- 用户可修复的输入错误、管理员需修复的服务配置、外部 provider 业务拒绝和暂时不可用故障必须使用不同错误类别与文案。
- 无法在前端判断的失败仍应返回稳定、可消费的错误码和脱敏摘要，不能只返回通用内部错误。

## 三、明确能力与责任边界

- 相似功能若依赖不同资源，必须独立表达其前置条件；例如 ACME DNS-01 的服务端 Token 不等同于 Cloudflare Origin CA credential。
- 操作者不能直接修复的条件应明确标注为服务器管理员操作，不应伪装成表单字段或让用户反复重试。
- 需要外部平台配合的操作应说明最小权限、预期传播时间和验证方式，但不在 UI 中保存无关或敏感的外部凭据。
- 已知产品边界应直接说明，不以不可用按钮或沉默失败暗示功能已经支持。

## 四、失败不丢失上下文

- 失败后保留安全的表单输入、筛选条件、返回路径和待完成任务，支持修复后直接重试。
- 用户主动取消、页面刷新或会话过期时，应明确区分未提交、提交中、已成功和结果未知，避免重复提交高风险操作。
- 对异步操作展示开始时间、最近尝试、下一次尝试、失败次数和可重试性；不要让操作者猜测后台是否仍在执行。
- 可复制诊断信息应仅包含资源标识、错误码、时间和脱敏摘要，便于支持排查而不泄露敏感材料。

## 五、安全可见，而非安全沉默

- Token、私钥、Cookie secret、一次性激活链接和完整第三方响应不得进入页面、日志、审计、错误消息、浏览器存储或测试快照。
- 敏感字段使用 write-only 交互；查询仅显示状态、指纹、最后校验时间和脱敏错误。
- 安全限制必须解释其目的和影响，例如“需要强确认”“仅 Cloudflare 回源可用”或“未认证请求被拒绝”，不能只静默禁用。
- 高风险操作必须要求与资源绑定的显式确认，不以预填字段、一次点击或隐藏状态替代确认语义。

## 六、默认保护可用服务

- 首次创建失败不得产生会被运行时选中的坏状态或坏材料。
- 续期、轮换、配置更新和 provider 同步失败时，优先保留经过验证的现有可用状态。
- 删除、解绑、停用和撤销前，应展示受影响资源与服务中断范围；执行结果应明确哪些资源已改变、哪些仍需处理。
- 对运行时无法安全服务的状态必须 fail closed，并在管理面给出恢复指引。

## 七、渐进披露复杂度

- 列表优先展示身份、当前可服务性、风险和最近异常；高级元数据放在详情、展开区或工具提示中。
- 默认表单只要求完成当前任务所需字段，高级配置应有明确名称、默认值来源和影响说明。
- 首次使用提供空状态引导；筛选无结果、无权限和系统错误必须使用不同空状态。
- 不用技术术语替代用户意图；必须保留技术术语时，同时提供面向任务的解释。

## 八、一致性与可访问性

- 同类动作使用一致的位置、名称、状态颜色和确认方式；禁用原因应可通过键盘和辅助技术读取。
- 表单字段必须有可关联的标签、错误信息和焦点顺序；错误提交后焦点移动到第一个可修复问题。
- 状态不可只依赖颜色，应同时提供文字、图标或结构化标签。
- 桌面和移动端均应保留核心状态、确认和错误恢复能力；不得因屏幕宽度隐藏安全关键提示。

## 九、验收与评审

涉及管理面或运行时状态的 Change 应至少回答：

- 操作者在提交前能否知道操作是否可执行？
- 失败后能否区分输入、配置、权限和 provider 故障？
- 错误是否给出安全且可执行的恢复路径？
- 高风险操作是否明确展示影响范围并要求合适确认？
- 异步或失败场景是否保留上下文与可观察状态？
- 是否验证了键盘访问、状态文字化和窄屏下的关键流程？
- 是否确认 API、日志、审计、UI 和测试中均未泄露敏感数据？

## 使用方式

- 新需求在 `requirements/` 中把相关原则转化为可验证的交互和验收口径。
- Change 在设计与验收章节引用本文件，并记录本次原则的具体落实方式。
- 发现跨功能、长期有效的新约束时更新本文件；单一功能的局部细节留在对应需求文档。
