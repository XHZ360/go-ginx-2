# Change 文档工作流

Change 文档用于管理一次有明确目标和结束条件的改动。一个 Change 只使用一个 Markdown 文件，集中记录背景、目标、设计、实施、验证与结果，不再拆分 proposal、design、tasks、specs 等多套模板。

## 何时创建

满足任一条件时应创建 Change：

- 改变产品行为、公开 API、数据模型、协议或安全边界。
- 跨越多个模块，无法通过一个局部提交完整说明。
- 需要数据迁移、兼容期、灰度、回滚或远端资源处理。
- 当前实现与目标模型存在偏差，需要分阶段收敛。
- 用户明确要求先形成方案、计划或实施记录。

仅修正文案、拼写、失效链接或局部实现细节时，通常不需要 Change。

## 目录与状态

| 目录 | 状态 | 含义 |
| --- | --- | --- |
| `changes/active/` | `active` | 已接受、正在设计或实施，尚未完成 |
| `changes/completed/` | `completed` | 已实现并验证，最终事实已同步到长期文档 |
| `changes/archive/` | `archived` | 已过期、被替代、被否决或仅保留历史上下文 |

文件名使用稳定的 kebab-case 主题名，例如 `domain-path-proxy-routing.md`。状态变化通过 `git mv` 移动同一个文件，不复制新版本。

## 事实边界

- `active` Change 描述目标和计划，不代表代码已经实现。
- 当前已实现行为仍以代码、测试和 `architecture/` 为准。
- 产品目标与验收口径维护在 `requirements/`；Change 只链接并记录本次差异。
- 长期且难以回退的选择记录到 `decisions/`；Change 链接该决策，不重复维护决策正文。
- 完成 Change 时必须把最终行为合并到 `requirements/`、`architecture/` 或 `operations/`，不能让 `completed/` 成为唯一事实来源。

当目标模型与当前实现不同，必须分别使用“当前实现”和“目标状态”章节，禁止使用容易误判为已完成的现在时表述。

## 生命周期

### 1. 建立

从 [change-template.md](../changes/change-template.md) 创建 `changes/active/<change-name>.md`：

1. 填写状态、最后更新日期和相关文档。
2. 写清当前问题、目标和非目标。
3. 列出代码或测试证据，避免根据旧文档猜测现状。
4. 标出尚未决定的问题；阻塞问题未关闭前不得进入实施。

### 2. 设计评审

开始实施前，Change 至少应明确：

- 核心不变量和验收条件。
- 数据模型、运行时流程和错误语义。
- API/UI/协议影响（无影响时明确写无）。
- 安全、敏感数据和失败恢复边界。
- 兼容、迁移、回滚和数据冲突处理。
- 最小验证范围。

影响长期方向的选择先形成 `decisions/` 文档，再在 Change 中引用。

### 3. 实施

- 按实施顺序维护任务清单，完成后立即勾选。
- 新发现的范围、风险和决定直接回写同一文件。
- 不把聊天记录、逐日流水或完整日志粘贴进 Change。
- 实际实现偏离设计时，先更新 Change 和相关决策，再继续实施。
- `worklog.md` 只保留当前进度摘要，并链接 active Change。

### 4. 验证与完成

移入 `completed/` 前必须满足：

- 实现和必要迁移已完成。
- 验收条件逐项有结果。
- 记录实际执行的测试/构建命令及结果；未执行项明确说明原因。
- requirements、architecture、operations 与 Admin UI 文档已同步。
- 已检索旧路径、旧字段和被替代模型的残留描述。
- `worklog.md` 已更新当前状态与下一步。

完成时把状态改为 `completed`，填写完成日期、实现提交和验证结果，然后使用 `git mv` 移到 `changes/completed/`。

### 5. 归档

以下情况移入 `archive/`：

- 方案被否决或取消。
- 后续 Change/Decision 已取代该文档。
- completed 记录已无日常参考价值，只需保留历史上下文。

归档文档顶部必须说明“不代表当前状态”，并链接替代文档或当前事实来源。

## 状态门槛

### Ready for Implementation

- [ ] 问题、目标和非目标清晰
- [ ] 阻塞决策已关闭
- [ ] 当前实现有代码/测试证据
- [ ] 目标模型和核心不变量完整
- [ ] 迁移、兼容和回滚策略明确
- [ ] 验收条件与验证命令明确

### Ready for Completion

- [ ] 实施任务全部完成或明确取消
- [ ] 验收条件全部有结论
- [ ] 实际验证结果已记录
- [ ] 长期事实文档已同步
- [ ] 旧模型与旧路径残留已检索
- [ ] worklog 已更新

## 维护要求

- 元信息中的日期、状态和提交必须反映实际情况。
- 不在 Change 中保存令牌、凭据、私钥、完整敏感响应或生产数据。
- 一个事实只保留一个主来源；Change 使用相对链接引用。
- 纯文档阶段不得把目标架构写成当前已实现行为。
- Change 完成后仍发现模型错误时，新建 active Change 纠正，并把旧 completed 文档标记为被替代；不要直接改写历史使其看似从未发生。
