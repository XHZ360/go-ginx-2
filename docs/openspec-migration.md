# OpenSpec 迁移说明

## 目标

将 `openspec/` 中仍然有效的项目知识迁移到普通 Markdown 文档，之后以代码、测试和 `docs/` 描述项目当前状态，不再维护 proposal、design、spec、tasks 等固定工作流。

迁移不是逐文件改名。应删除重复、过期和仅服务于实施过程的内容，只保留使用者或维护者仍需了解的信息。

## 内容去向

| OpenSpec 内容 | 处理方式 |
| --- | --- |
| `openspec/specs/` 当前规格 | 按主题整理进 `docs/`，描述当前行为、边界和验证入口 |
| 尚未完成的 change | 完成前保留；完成后把最终行为并入对应主题文档 |
| change 中仍有效的技术决策 | 合并到架构文档的“关键决策”部分；确有独立价值时建立简短决策记录 |
| proposal、tasks 和已完成计划 | 不迁移，Git 历史已经保存实施过程 |
| WHEN/THEN 场景 | 提炼为行为说明、边界条件或测试入口，不机械保留格式 |
| 已过期、重复或无法从代码验证的描述 | 删除；必要时先建立待确认事项，不把猜测写成当前事实 |

## 建议的文档结构

目录按读者需要组织，不要求与代码包或原 capability 一一对应：

```text
docs/
  architecture.md                 系统组成、进程关系和数据流
  control-channel.md              认证、会话、协议和流生命周期
  reverse-proxy.md                TCP/UDP/HTTP/HTTPS 当前行为
  certificate-management.md       证书来源、状态、续期和失败语义
  admin-and-observability.md       Admin API/UI、审计和指标
  deployment.md                   配置、部署、升级、回滚和排障
  limits.md                       已执行的限制及明确的已知缺口
  decisions/                      少量仍影响当前实现的重要决策（按需建立）
```

这只是初始分类。已有 `daemon-runtime.md`、`docker-development.md` 等文档内容准确时应直接扩充，不要另建重复文档。

## 迁移步骤

### 1. 冻结 OpenSpec

- 不再创建新的 change 或补齐旧模板。
- `openspec/` 在迁移期间只读；进行中的 `add-goginx-sdk` 可继续保留到功能稳定。
- 新需求直接修改代码、测试及确实受影响的 `docs/` 文档。

### 2. 建立当前事实

按以下顺序确认内容：

1. 运行中的代码和公开接口。
2. 自动化测试表达的行为。
3. 当前 README 和 `docs/`。
4. `openspec/specs/`。
5. 历史 change。

来源冲突时，以可运行行为为基准，并判断这是文档过期还是实现缺陷。不要为了匹配旧规格直接改变代码。

### 3. 按主题提炼

每次迁移一个主题，例如先处理证书，再处理控制通道。主题文档通常只需要：

- 用途和范围；
- 当前工作方式或关键数据流；
- 对使用者重要的配置、状态和错误语义；
- 安全、兼容性及失败边界；
- 相关命令、代码入口或测试入口。

章节可以按内容自由组织。代码已经清楚表达、且读者不需要预先知道的内部细节不必复制到文档。

### 4. 审核与去重

- 检查 README、已有 `docs/` 和新文档是否重复；保留一个事实来源，其他位置改为链接。
- 确认文档描述的是已实现行为，并给出可定位的代码或测试入口。
- 将未实现设想明确标成“已知缺口”或移除，不能与当前能力混写。
- 不迁移完成状态复选框、阶段计划、临时讨论和已经失效的替代方案。

### 5. 切换入口

- 更新 README 的文档索引和仓库结构说明。
- 保持 `AGENTS.md` 只包含长期协作规则，不把产品规格搬进去。
- 搜索并替换代码、脚本和文档中仍指向 `openspec/` 的有效链接。

### 6. 删除 OpenSpec

满足以下条件后，在一个独立变更中删除整个 `openspec/`：

- 10 个当前 capability 的有效信息均已迁移、合并或明确判定为过期；
- `add-goginx-sdk` 的最终行为已经进入代码、测试和相应文档；
- README 和其他文档不再把 OpenSpec 当作当前事实来源；
- 仓库中没有工具或 CI 依赖 OpenSpec；
- 关键运行、部署、安全和故障处理信息可以从 `docs/` 找到。

删除前无需把归档 change 搬进 `docs/history/`；Git 历史就是旧方案和实施记录的存档。

## 推荐迁移顺序

1. 证书管理、ACME 和生命周期健康：风险最高，并已有多份相关文档可合并。
2. 控制通道和反向代理运行时：项目核心架构。
3. 部署、Admin、可观测性和审计：面向操作者的知识。
4. 配额与限制：区分真实执行的限制和未来设想。
5. 文档对齐类规格：吸收有效内容后直接删除，不建立对应主题文档。

每完成一个主题即可从迁移清单中划掉，不需要等待所有文档一次性重写。
